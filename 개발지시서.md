## 0. 개발 원칙

- **전체 플로우:** `개발 → 더미데이터 투입 → QA → 엑셀 데이터 마이그레이션(원본→표준화→최종수정본) → 완성`
- **한 번에 전부 만들지 말 것.** 아래 "구현 순서"에 명시된 단계별로 PR/커밋을 나눠서 진행하고, 각 단계가 끝나면 확인 후 다음 단계로 넘어감
- **실 데이터는 QA 통과 전까지 절대 투입하지 않는다.** 개발/QA 중에는 더미데이터만 사용. 엑셀 원본 데이터 이관은 **QA 완료 후 맨 마지막 단계**에서 진행
- **데이터 마이그레이션은 3단계로 진행:** (1) 엑셀 원본 값 → (2) 시스템이 제안하는 표준화 값 → (3) 사용자가 최종 수정한 값. 세 단계 모두 화면에 나란히 표시하고 사용자 컨펌 후에만 DB에 최종 입력
- **복잡한 비즈니스 로직은 반드시 단위 테스트로 검증.** 특히 상태값 자동 판정 로직과 정산 계산 로직
- **불확실한 사양은 추측하지 말고 질문할 것.** 이 문서에 명시되지 않은 사항은 확인 후 구현

## 1. 프로젝트 개요

### 1.1 목적
베트남 교육생(요양보호사 지망)을 한국의 교육원에 매칭하고, 자격증 취득 후 요양원 취업 및 비자 변경까지 연결하는 업무 전반을 엑셀에서 웹 관리자 페이지로 이관.

### 1.2 사용자
내부 업무 담당자 2~3명. 매일 사용. 외부 공개 X.

### 1.3 비즈니스 프로세스

```
1. 고객 상담 → 기본정보 취득
2. 희망 조건에 맞는 교육원 매칭 (없으면 발굴)
3. 교육 예약금 입금 (35,000원 단독 / 100,000원 = 교육+웰컴팩)
4. 교육원에서 수강
5. 자격증 시험 응시 및 합격
6. 요양보호사 자격증 취득
7. (선택) 웰컴팩 예약금 입금
8. 요양원 면접 연결
9. 면접 합격
10. E-7-2 비자 변경
```

### 1.4 핵심 도메인 용어
- **교육생 (고객)**: 베트남인, 요양보호사 자격증 취득 희망자
- **교육원**: 요양보호사 교육 제공 기관 (한국)
- **요양원**: 교육생의 취업처
- **웰컴팩**: 자격증 취득자에게 요양원 취업을 소개하는 유료 서비스 (정가 150만원)
- **소개비**: 교육생 1인당 교육원이 우리에게 지급하는 수수료 (교육비의 25%)

## 2. 기술 스택

### 2.1 결정된 구성
- **프론트엔드**: Next.js 14+ (App Router) + TypeScript
- **UI 라이브러리**: Tailwind CSS + shadcn/ui
- **백엔드/DB**: Supabase (PostgreSQL + Auth + Realtime)
- **배포**: Vercel (도메인 없음, `*.vercel.app` 사용)
- **데이터 페칭**: Supabase JS Client + TanStack Query
- **폼**: React Hook Form + Zod
- **날짜 처리**: date-fns
- **엑셀 파싱** (이관 시): xlsx 또는 SheetJS

### 2.2 환경 변수 (.env.local)
```
NEXT_PUBLIC_SUPABASE_URL=
NEXT_PUBLIC_SUPABASE_ANON_KEY=
SUPABASE_SERVICE_ROLE_KEY=    # 서버 사이드에서만 사용, 클라이언트 노출 금지
GOOGLE_TRANSLATE_API_KEY=     # 베트남어 → 한국어 번역용

# NHN Cloud SMS (Simple/Notification Service)
NHN_SMS_APP_KEY=
NHN_SMS_SECRET_KEY=
NHN_SMS_SEND_NO=              # 발신번호 (사전등록 필요)
```

**NHN Cloud SMS 연동 참고:**
- 엔드포인트: `https://api-sms.cloud.toast.com/sms/v3.0/appKeys/{appKey}/sender/sms`
- 인증: 헤더에 `X-Secret-Key`
- Next.js **API Route (`/api/sms/send`)** 에서만 호출. 키는 클라이언트 노출 금지.

### 4.2 주요 테이블 상세

#### 4.2.1 `customers` (교육생)
```sql

created_at            timestamptz (최초 등록일 = 엑셀의 "업데이트" 컬럼 아님, 신규 필드)

legacy_status         text (엑셀의 기존 상태값, 참고용으로만 보존)

-- 개인정보
name_vi               text            -- 베트남 이름
name_kr               text            -- 한국 이름
address               text
gender                text            -- '여' / '남'
birth_year            integer
phone                 text
email                 text

-- 비자
visa_type             text            -- D-10, F-2-R, D-2, 귀화 등
topik_level           text            -- TOPIK 급수 (예: '3', 'KIIP 4')

                                      -- 표준화가 어려운 필드, 일단 text로 둠

-- 희망 교육 조건
desired_period        text            -- "7월", "9월초", "2026/3월" 등 자유형식
desired_time          text            -- '주간' / '야간'
desired_region        text

-- 매칭된 교육원/요양원
training_center_id    uuid FK (nullable) -- training_centers.id
training_class_id     uuid FK (nullable) -- training_classes.id
care_home_id          uuid FK (nullable) -- care_homes.id

-- 교육/취업 일정
class_start_date      date (nullable) -- 강의 시작일
class_end_date        date (nullable) -- 강의 종료일
work_start_date       date (nullable) -- 근무 시작일
work_end_date         date (nullable) -- 근무 종료일
visa_change_date      date (nullable) -- 비자 변경일
interview_date        date (nullable) -- 면접일

-- 상품
product_type          text            -- '교육' / '웰컴팩' / '교육+웰컴팩' / null

-- 대기 정보
is_waiting            boolean DEFAULT false
recontact_date        date (nullable)
waiting_memo          text (max 500)

-- 종료 상태 (수동)

```

#### 4.2.2 `customer_statuses` (진행 단계 체크)
고객의 진행 단계를 세분화하여 각 단계별 완료 여부를 관리. 기존 "1-4.접수포기" 같은 단일 상태값 대신 이 테이블로 체크.

```sql
customer_id                    uuid PK FK -> customers.id (1:1 관계)

-- 접수 단계 (수동 플래그)
intake_abandoned               boolean DEFAULT false  -- 접수포기 (수동)
study_abroad_consultation      boolean DEFAULT false  -- 유학상담 (수동)

-- 교육 예약 단계 (수동 플래그)
training_center_finding        boolean DEFAULT false  -- 교육원 발굴 (수동)
training_reservation_abandoned boolean DEFAULT false  -- 교육 예약포기 (수동)

-- 교육 단계 (수동 플래그)
certificate_acquired           boolean DEFAULT false  -- 자격증 취득 (수동)
training_dropped               boolean DEFAULT false  -- 교육 드랍 (수동) — 교육 중 이탈

-- 취업 단계 (수동 플래그)
welcome_pack_abandoned         boolean DEFAULT false  -- 웰컴팩 예약포기 (수동)
care_home_finding              boolean DEFAULT false  -- 요양원 발굴 (수동)
interview_passed               boolean DEFAULT false  -- 면접 합격 (수동)

updated_at                     timestamptz
```

**완전 자동 계산 필드는 DB에 저장하지 않고 애플리케이션 레이어에서 계산하거나 View/RLS 함수로 노출.** (예: `intake_complete`, `training_reservation_complete`, `employment_complete` 등)

#### 4.2.3 `customer_consultations` (상담 일지)

엑셀의 기존 단일 상태값 목록. 편집 가능해야 하고, 삭제 시 기존 사용 레코드를 다른 값 또는 null로 이행.
```sql
id                  uuid PK
code                text UNIQUE   -- '1-1.단순문의' 등
label               text
display_order       integer
```

## 5. 핵심 비즈니스 로직 (매우 중요)

### 5.1 상태 단계 자동 판정 로직

각 고객의 진행 상태는 아래 "핵심 체크포인트"로 표시된다. 체크포인트는 **수동 플래그(customer_statuses 테이블에 저장)**, **완전 자동(런타임 계산)** 두 종류.

#### 5.1.1 접수 단계
| 체크포인트 | 타입 | 판정 로직 |
|---|---|---|
| 기초정보 수집 | 완전자동 | 개인정보(name_vi, name_kr, phone, address, gender, birth_year, visa_type) **모두 입력** = '완벽' / name_kr 또는 name_vi 중 하나 & phone **둘 다 입력** = '핵심' / 이름+전화 중 하나라도 없음 = '없음' |
| 접수포기 | 수동 bool | `customer_statuses.intake_abandoned` — true이면 **이후 모든 단계 판정 중지** |
| 유학상담 | 수동 bool | `customer_statuses.study_abroad_consultation` — true이면 **이후 모든 단계 판정 중지** |
| **접수 완료** | 완전자동 | 기초정보 ∈ {'핵심','완벽'} AND NOT 접수포기 AND NOT 유학상담 |

#### 5.1.2 교육 예약 단계
| 체크포인트 | 타입 | 판정 로직 |
|---|---|---|
| 교육원 발굴 | 수동 bool | `customer_statuses.training_center_finding` — **교육원 매칭이 yes로 바뀌면 자동으로 false로 리셋**. true일 때는 교육원 매칭/강의일정 확정 영역 UI 비활성화 |
| 교육원 매칭 | 완전자동 | `customers.training_center_id IS NOT NULL` |
| 강의일정 확정 | 완전자동 | `customers.training_class_id IS NOT NULL` |
| 교육 예약금 입금 | 완전자동 | 해당 고객의 `reservation_payments` 중 `payment_date IS NOT NULL`인 레코드 존재 |
| 교육 예약포기 | 수동 bool | `customer_statuses.training_reservation_abandoned` — true이면 이후 단계 중지 |
| 강의 접수 메시지 발송 | 완전자동 | `sms_messages` 중 `message_type='new_student' AND target_customer_id=해당 고객` 레코드 존재 |
| **교육 예약 완료** | 완전자동 | NOT 교육원발굴 AND NOT 교육 예약포기 AND 교육원 매칭 AND 강의일정 확정 AND 예약금 입금 |

#### 5.1.3 교육 단계
| 체크포인트 | 타입 | 판정 로직 |
|---|---|---|
| 메시지 발송 | 완전자동 | 교육 예약 단계와 동일 |
| 교육 전/중/완료 | 완전자동 | `class_start_date`, `class_end_date` 기준. 오늘이 `class_start_date` 이전 = '전', 사이 = '중', 이후 = '완료'. 강의일정 정보 없으면 비활성화 |
| 교육 드랍 | 수동 bool | `customer_statuses.training_dropped` — 교육 중 이탈. true이면 이후 모든 단계 판정 중지 |
| 자격증 취득 | 수동 bool | `customer_statuses.certificate_acquired` |
| **교육 완료** | 완전자동 | NOT 교육 드랍 AND 메시지발송 AND 교육='완료' AND 자격증 취득 |

> **이탈 위치 추적 원칙**: 단계별 이탈 플래그가 단계별로 존재해야 함. 현재 구조: 접수(intake_abandoned, study_abroad_consultation) / 예약(training_reservation_abandoned) / 교육(training_dropped) / 취업(welcome_pack_abandoned) / 종료(termination_reason).

#### 5.1.5 근무 단계
| 체크포인트 | 타입 | 판정 로직 |
|---|---|---|
| 근무 전/중/종료 | 완전자동 | `work_start_date` 이전 = '전' / 이후 AND `work_end_date IS NULL` = '중' / `work_end_date IS NOT NULL` = '종료' |

### 5.2 정산 상태 요약 로직
특정 교육생의 정산 현황은 4가지 항목으로 요약:
- **예약금 입금**: 완료 / 미완료 — `reservation_payments.payment_date IS NOT NULL`
- **소개비 정산**: 완료 / 미완료 — `commission_payments.status = 'completed'` (입금 + 세금계산서 모두 완료)
- **이벤트 정산**: 완료 / 미완료 / 대상아님 — 해당 고객에게 `event_payments` 레코드가 없으면 '대상아님', 있으면 `gift_given` 기준
- **웰컴팩 정산**: 완료 / 미완료 / 대상아님 — 상품이 웰컴팩/교육+웰컴팩이면 대상, 그 외 '대상아님'. `sales_reported = true`이면 완료

### 5.3 소개비 정산 대상 선정
- 매월 **말일 기준**
- **주간반 교육생**: `class_start_date + 45일`이 포함된 월의 교육생이 그 달의 정산 대상
- **야간반 교육생**: `class_start_date + 75일`이 포함된 월의 교육생이 그 달의 정산 대상
- 예 (주간): 3월 15일 개강 → 3월 15일 + 45일 = 4월 29일 → 4월 정산 대상
- 예 (야간): 2월 15일 개강 → 2월 15일 + 75일 = 5월 1일 → 5월 정산 대상
- 주간/야간 구분은 `training_classes.class_type`으로 판단 (`'weekday'` / `'night'`)

### 5.4 예약금 처리 로직
35,000원 예약금은 4가지 중 하나로 처리 (드롭다운 선택):
1. **중도 탈락 → 매출 인식**: 교육생이 중도 포기, 예약금은 우리 매출. 소개비 공제 없음.
2. **교육생 환급 (공제 없음)**: 예약금을 교육생에게 환불. 소개비에서 공제 없음.
3. **소개비 공제**: 소개비 정산 시 예약금 35,000원만큼 공제. (교육원으로부터 받는 금액이 줄어듦)
4. **교육원 섭외 실패 → 환불**: 교육원 매칭 실패로 환불.

웰컴팩 예약자(100,000원)는 **공제 처리 대상이 아님** (소개비에서 차감 안 함).

### 5.5 웰컴팩 결제 계산
```
최종 결제 금액 = welcome_pack_price - discount_amount

1회차: reservation_amount (예약금) = 100,000 (첫 예약 시) 또는 0 (나중에 예약)
2회차: interim_amount (잔금1) = 교육원 지역별 250,000 / 300,000 / 350,000
3회차: balance_amount (잔금2) = 최종 결제 금액 - reservation - interim
```

**잔금1 (2회차) 지역별 매핑 기준 (교육원 소재지 기준):**
- **250,000원**: 서울에서 대중교통(지하철/버스)으로 이동 가능한 거리 — 서울/경기/인천 등 수도권
- **300,000원**: KTX 일반석 편도 정가 **4만원 이하**인 지역 — 대전/충남/충북 등
- **350,000원**: KTX 일반석 편도 정가 **4만원 초과** 지역 — 부산/대구/광주/경남/경북/전남/전북 등

UI 구현 시: 교육원 지역 정보를 기준으로 **추천 금액을 기본값으로 자동 선택**하되, 드롭다운으로 수동 변경 가능. 교육원이 아직 매칭 안 된 경우는 수동 선택만 가능.

예시: 교육+웰컴팩 첫 예약 (총 1,200,000원 = 1,500,000 - 300,000)
- 1회차 100,000원 (예약금)
- 2회차 300,000원 (잔금1, 충남 교육원 기준)
- 3회차 800,000원 (잔금2)

  - 7-2. 교육원 발굴: `customer_statuses.training_center_finding = true`

  - 7-6. 강의 접수 메시지 발송: 접수 단계 완료 AND **강의 접수 메시지 발송 no**
  - 7-7. 요양원 발굴: `customer_statuses.care_home_finding = true`
  - 7-8. 비자 변경: 비자변경 상태가 '대기' 또는 '중'

탭 구조:
1. **기본 정보** 탭

   - 비자 섹션: 비자, TOPIK, 체류 남은기간

4. **정산** 탭
   - 상단에 요약 (4개 상태)
   - 예약 결제 카드 / 소개비 결제 카드 / 이벤트 결제 카드 / 웰컴팩 결제 카드
   - 각 카드에서 인라인 편집 가능

#### 6.3.3 신규 고객 등록 화면
- 2단 폼: 좌측 개인정보, 우측 비자/희망조건
- 코드는 서버에서 자동 발급 (`CVN` + `YYMM` + 순번 3자리)

### 6.4 교육원 (`/training-centers`)
- 목록: 코드, 지역, 이름, 원장, 연락처, 수강료, 교육생 수 (계산된 값)
- 상세: 기본 정보 + 월별 개강 정보 (테이블) + 해당 교육원 소속 교육생 목록

### 6.5 요양원 (`/care-homes`)
- 목록: 코드, 이름, 지역, 담당자, 베드, 특이사항
- 상세: 기본 정보 + 소속 교육생 (등록/구직/취업 중 분류)

### 6.6 정산 (`/settlements`)

#### 6.6.1 조회 방식
두 가지 뷰를 탭으로 전환:
- **교육생별 뷰**: 고객 검색 → 해당 고객의 4종 정산 상태 요약 + 상세
- **교육원별 뷰**: 교육원 검색 → 해당 교육원 소속 교육생들의 정산 현황 테이블

#### 6.6.2 요약 카드
```
┌─────────────────────────────────────┐
│ [교육생명]                            │
│ 예약금: ✓완료 | 소개비: ✗미완료       │
│ 이벤트: 대상아님 | 웰컴팩: ✓완료       │
└─────────────────────────────────────┘
```

#### 6.6.3 소개비 정산 워크플로우
1. 매월 말, 정산 대상 고객 자동 추출
2. 교육원별로 그룹화하여 테이블 표시
3. "SMS 발송" 버튼 → 알림발송 화면으로 이동
4. SMS 발송 후: 입금 완료/일자, 세금계산서 발행 완료/일자 체크 UI 제공
5. 두 항목 모두 완료 시 `status='completed'`로 전환

#### 6.6.4 예약금 처리 드롭다운
예약 결제 레코드의 `refund_reason` 변경 시:
- '소개비 공제' 선택 → 해당 고객의 소개비 정산 시 자동 공제 로직 적용
- '교육생환급' 선택 → 공제 없음
- 기타 옵션도 UI에 반영

### 6.7 알림발송 (`/sms`)

#### 6.7.1 신규 교육생 알림 (`/sms/new-student`)
- 발송 대상 목록: 교육 대기중 ~ 그 이후 단계 고객
- 각 행에 "발송" / "재발송" 버튼
- 메시지 미리보기 모달 제공 (엑셀의 함수를 컨버팅한 템플릿 사용)
- 발송 시 `sms_messages` 테이블에 기록
- 블라블라/특이사항은 직접 입력, 마지막 입력값 자동완성

#### 6.7.2 수수료 정산 알림 (`/sms/commission`)
- 매월 말 기준 정산 대상 교육생을 교육원별로 그룹화
- 각 교육원 카드에 해당 교육생 목록 + 정산액 요약 + "SMS 발송" 버튼
- 예약금 공제 여부가 교육생마다 다를 수 있으므로 **각 교육생의 예약금 처리 상태를 반영한 최종 정산액**을 교육원에 보내는 메시지에 포함
- 메시지 템플릿은 엑셀의 `5.메시지` 시트 참고하여 변환

### 6.8 설정 (`/settings`)

#### 6.8.1 상태값 관리
- 기존 `status_options` 목록 (엑셀의 "1-1.단순문의" 등)
- 추가/수정/삭제 가능
- 삭제 시: 해당 상태를 쓰는 고객의 `legacy_status`를 "다른 값으로 변경" 또는 "(값없음)"으로 이행하는 선택 다이얼로그

#### 6.8.2 결제 설정
편집 가능한 값들:
- 교육 예약금 (기본 35,000)
- 웰컴팩 예약금 (기본 100,000)
- 웰컴팩 정가 (기본 1,500,000)
- 웰컴팩 예약 할인 (기본 300,000)
- 웰컴팩 잔금 옵션 (드롭다운 옵션: [250,000, 300,000, 350,000] 기본)
- 소개비 비율 (기본 0.25 = 25%)
- 이벤트 종류 옵션 (드롭다운 편집 가능)
- 이벤트 상품권 종류 옵션 (드롭다운 편집 가능)

#### 6.8.3 계정 관리
- 초기 계정 **3개** 생성 (권한 차등 없음, 모두 동일 권한)
- 외부 공개 회원가입 기능 **없음**. 공개 `/signup` 페이지 만들지 말 것
- 로그인한 사용자가 설정 페이지에서 **신규 계정 생성 가능**
  - 이메일 + 임시 비밀번호 입력 → Supabase Auth Admin API로 사용자 생성
  - Admin API는 `service_role` 키 필요 → **Next.js API Route (`/api/admin/users/create`)** 에서만 호출, 클라이언트 노출 금지
- 비밀번호 변경: 본인 계정에 한해 설정 페이지에서 변경 가능 (Supabase `auth.updateUser({password})` 사용)
- 계정 목록 조회 / 비활성화 기능 포함
- 계정 삭제는 Supabase 대시보드에서 직접 처리 (UI로는 제공하지 않음)

## 7. 번역 기능 (베트남어 → 한국어)

상담 일지 작성 시:
- 사용자가 베트남어로 입력
- "AI로 번역" 버튼 클릭 → 번역 API 호출 (Google Translate API 또는 DeepL)
- 번역된 한국어와 원문 베트남어 모두 저장
- 고객 상세에서는 한국어 + "원문 보기" 토글로 베트남어 노출

**API 선택 권장**: 비용/품질 모두 고려 시 Google Translate API. DeepL은 베트남어 지원이 제한적.
구현은 Next.js API Route로 프록시하여 API 키를 클라이언트에 노출하지 않음.

## 8. 인증 (Supabase Auth)

- **로그인 방식**: 이메일 + 비밀번호
- **초기 계정 3개**, 권한 차등 없음 (전원 동일 기능 접근 가능)
- **공개 회원가입 기능 없음** — `/signup` 같은 공개 라우트를 만들지 말 것
- **계정 생성은 로그인한 사용자가 설정 페이지에서만 가능** (6.8.3절 참고)
- 모든 페이지는 미들웨어로 보호 (`middleware.ts`)
- 미로그인 상태 → `/login` 으로 리다이렉트
- 비밀번호 변경: 설정 페이지에서 본인 계정에 한해 가능

## 10. 데이터 표준화 규칙 (Phase 10 상세)

엑셀 데이터를 DB로 넣기 전에 다음 규칙을 적용하고 **사용자에게 원본 vs 표준화 비교표를 보여준 뒤 컨펌을 받음**:

| 필드 | 원본 예시 | 표준화 규칙 |
|---|---|---|
| 코드 | "CVN2508001" | 공백 제거, 대문자화 |
| 이름(한/베) | "팜 티 중" / " Phạm Thị Dung " | 앞뒤 공백 제거, 다중 공백 단일 공백으로 |
| 전화번호 | "010-7240-6777", "010 7240 6777", " 010-4469-9612" | 공백/하이픈 제거 후 `010-XXXX-XXXX` 포맷으로 통일 |

## 11. 기획자 확인사항 (v1.1에서 모두 해결됨)

v1.0 초안에서 모호했던 항목들의 확정 결론:

1. **단계별 이탈 플래그 원칙** — 각 단계마다 이탈 위치를 추적. 교육 드랍 플래그(`training_dropped`) 추가 (5.1.3절).
2. **4-5.취업포기** → `termination_reason = '요양보호사 직종변경'` 로 매핑.
3. **4-6.자체취업** → 별도 플래그 없음. 상품이 `교육`(웰컴팩 미구매)인 고객으로 남음.
4. **야간반 소개비 정산** → `class_start_date + 75일` 기준 (주간은 45일).
5. **대쉬보드 7-5/7-6** → 7-5는 예약금 입금 no, 7-6은 강의 접수 메시지 발송 no로 확정.
6. **웰컴팩 잔금 지역 매핑** → 25만원(서울 대중교통 가능권) / 30만원(KTX 편도 4만원 이하) / 35만원(KTX 편도 4만원 초과). 교육원 지역 기준 자동 추천 + 수동 변경 가능 (5.5절).
7. **SMS API** → NHN Cloud Simple/Notification Service 사용 (2.2절).
8. **로그인 계정** → 초기 3개, 권한 차등 없음. 설정 페이지에서 신규 계정 생성 및 비번 변경 가능 (6.8.3절).
9. **퍼포먼스 시트** → 자동 기록 기능 불필요. 구현하지 않음.
10. **To do list 시트** → 별도 기능으로 살리지 않음.
